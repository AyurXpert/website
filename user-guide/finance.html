1. Overview

This guide covers every step of the financial workflow in AyurXpert HMS — from registering the patient and collecting the registration fee, through pharmacy dispensing and IPD discharge settlement, to insurance claim lifecycle management, aging reports, and write-off approvals.

It is written for two audiences side-by-side:

  • Hospital staff (receptionists, accountants, pharmacists, nursing staff) — UI workflow descriptions with step-by-step instructions
  • System administrators and developers — database schema, SQL triggers, RLS patterns, constraint definitions, and technical behaviour of the system

Sections 2–7 are staff-facing. Section 8 is technical reference. Section 9 shows end-to-end scenarios. Section 10 is a troubleshooting table.

1.1 The 3-Tier Financial Model

Every financial transaction in AyurXpert flows through exactly three stages:

Tier 1 — Registration & Billing (reception.html / dispensaryPOS.html / ipd.html)
Patient arrives → payer type is selected → bill record is created in the bills table → visit token issued

Tier 2 — Claim Management (finance.html Insurance Cycles / insurance-claims.html)
Accountant fills TPA/policy details → pre-auth submitted → documents sent to insurer → claim status tracked

Tier 3 — Settlement & Close (ipd.html discharge panel / insurance-claims.html)
Insurer pays → approved amount entered → patient due recalculated → bill closed (status: paid / partial / settled)

For self-pay patients, only Tier 1 applies. For insurance and government scheme patients, all three tiers are used.

2. Setting Up Fee Structures

Fee structures must be configured before billing can work at reception. This is done in fee-admin.html by the dept_admin or super_admin.

2.1 Fee Categories

Each fee line item belongs to one of four categories:

CategoryWhen UsedExamples
Registration Charged once per new patient or per visit depending on hospital policy New Registration ₹50, Revisit ₹20
Consultation OPD doctor consultation fee, linked to OPD and doctor General OPD ₹200, Panchakarma OPD ₹300
Procedure Specific treatments or procedures performed in OPD/IPD Virechana ₹1,500, Basti ₹800
Package Bundled multi-session packages sold at the counter 7-day Panchakarma ₹12,000

2.2 Approval Workflow

No fee can be used for billing until it reaches active status. The approval workflow is a three-stage process:

accountant creates fee → pending
    ↓ dept_admin reviews and approves
dept_approved
    ↓ super_admin gives final activation
active ✓ — now visible at reception

Note: super_admin can skip the dept_approved step and activate directly from pending.
Warning — Fees not appearing at reception? Fees must reach active status before they appear in the billing calculator at reception. If a receptionist cannot see a fee in the dropdown, check fee-admin.html to see if the fee is still in pending or dept_approved state. Ask a super_admin to activate it.

Step-by-step: Create a new fee

  1. 1
    Open fee-admin.html — log in as accountant, dept_admin, or super_admin.
  2. 2
    Click "Add Fee" — a drawer or modal opens on the right side of the screen.
  3. 3
    Select the OPD — the fee is always linked to a specific OPD (e.g., General OPD, Panchakarma OPD).
  4. 4
    Select category — registration / consultation / procedure / package.
  5. 5
    Enter label and amount — e.g., label = "New Patient Consultation", amount = ₹250.
  6. 6
    Save — status is set to pending automatically. If you are a super_admin, you can directly click "Activate" to skip to step 8.
  7. 7
    Dept Admin reviews — logs into fee-admin.html, sees the pending fee in the list, clicks "Approve" to move status to dept_approved.
  8. 8
    Super Admin activates — sees the dept_approved fee, clicks "Activate". Status changes to active. Fee now appears in the billing calculator at reception for the linked OPD.

3. OPD Registration & Billing (reception.html)

All OPD billing starts at reception.html. The receptionist registers the patient, selects the payer type, and issues a visit token. The billing record (bills table row) is created at this step.

3.1 Self-Pay Workflow (most common)

The default payer type is Self-Pay. This covers all patients paying out of pocket — cash, UPI, or card.

  1. 1
    Search patient — enter phone number, ABHA number, or UHID in the search bar. Results appear immediately. Up to 6 family members sharing the same phone number will be shown.
  2. 2
    Select or create patient — click the correct name in the results list. If the patient is new, click "New Patient" and fill the registration form (name, age, gender, phone).
  3. 3
    Choose OPD and doctor — select the OPD from the dropdown (only active OPDs appear). Then select the doctor (only doctors marked active today for that OPD appear).
  4. 4
    Fee auto-populates — once the OPD is selected, the registration fee and consultation fee are filled automatically from the active fee structures. Review the amounts before proceeding.
  5. 5
    Payer Type toggle — the system shows four payer type buttons: 💵 Self-Pay · 🏥 Insurance/TPA · 🟢 PMJAY · 🔵 CGHS/ECHS. Self-Pay is selected by default. For a cash or UPI patient, leave it on Self-Pay.
  6. 6
    Payment mode and status — select Cash / UPI / Card. Then select payment status: Paid (if collected now) or Pending (if the patient will pay later). For most walk-in patients, select Paid.
  7. 7
    Click "Register & Create Visit" — a token number is issued (e.g., Token #12), the live queue on the doctor's screen updates, and a receipt can be printed. The bills and visits rows are created in the database at this moment.
Tip — Token number and queue The token number auto-increments within the day. It is reset to 1 at midnight. The doctor's screen (doctor.html) shows the waiting queue in real time — no page refresh needed.

3.2 Insurance / TPA Workflow

When the patient's treatment will be billed to an insurance company or TPA (Third Party Administrator), select the Insurance payer type at reception. The receptionist's role is simply to flag the visit as insurance — TPA details are filled by the accounts department later.

  1. 1
    Follow steps 1–4 from the Self-Pay workflow (search patient, select OPD/doctor, fees auto-populate).
  2. 2
    Click the 🏥 Insurance/TPA button in the Payer Type toggle row. The button highlights with a teal/blue colour.
  3. 3
    Automatic system behaviour: Payment mode switches to Credit/Due and is dimmed (the receptionist cannot override it). An advisory note appears: "Patient is insurance covered — TPA details to be filled by accounts department." Payment status is set to Pending.
  4. 4
    Register & Create Visit — click the button. The system saves payer_type = 'insurance' and insurance_claim_status = 'pre_auth_pending'. A token is issued. No payment is collected now.
Tip — Divide of responsibility The receptionist's job ends at step 4. Never ask the receptionist to fill TPA name, policy number, pre-auth number, or approved amounts — that is the accounts department's role, handled in finance.html Insurance Cycles → Edit modal.
Insurance patient returns after claim rejection? If a previously insurance-flagged patient's claim is rejected and they want to pay out of pocket, the accountant must update payer_type to self_pay and insurance_claim_status to not_applicable in the finance.html Insurance Cycles Edit modal before marking the bill as paid.

3.3 PMJAY Workflow

PMJAY (Pradhan Mantri Jan Arogya Yojana) patients are government scheme beneficiaries. The workflow is identical to Insurance/TPA but uses the dedicated PMJAY payer type.

  1. 1
    Follow steps 1–4 from Self-Pay (search, OPD, doctor, fees).
  2. 2
    Click the 🟢 PMJAY button in the Payer Type row. The system sets payer_type = 'pmjay' and insurance_claim_status = 'pre_auth_pending'.
  3. 3
    Payment mode → Credit/Due (auto). No cash collected. Token issued.
  4. 4
    Accountant later assigns an AYUSH HBP package code in insurance-claims.html → PMJAY Codes tab, gets Medical Officer approval (pmjay_mo_approved = true), and submits to the NHA PMJAY portal.
PMJAY eligibility verification AyurXpert does not automatically verify PMJAY eligibility (no NHA e-card API integrated yet). The receptionist must verify the patient's PMJAY e-card physically or via the NHA Beneficiary Identification System (BIS) portal before selecting the PMJAY payer type.

3.4 CGHS / ECHS Workflow

CGHS (Central Government Health Scheme) covers central government employees. ECHS (Ex-Servicemen Contributory Health Scheme) covers retired armed forces personnel. Both follow the same workflow.

  1. 1
    Follow steps 1–4 from Self-Pay.
  2. 2
    Click 🔵 CGHS/ECHS button. System sets payer_type = 'cghs' or 'echs' (the button label indicates which).
  3. 3
    For CGHS: collect the patient's CGHS Card number. For ECHS: collect the ESM Card number. Note these for the accountant — they fill the policy_number field in the Insurance Cycles Edit modal.
  4. 4
    Register & Create Visit. Token issued. No payment collected immediately.
ESI and Corporate payer types Two additional payer types exist in the database: esi (Employee State Insurance) and corporate (employer-contracted billing). These appear in the Finance dashboard KPI breakdowns but may not yet have dedicated UI buttons at reception in all builds. Check with your super_admin if these need to be activated.

3.5 What Gets Saved to the Database

When "Register & Create Visit" is clicked, the system creates one row each in the visits table and the bills table. For an insurance patient, the bill row looks like this:

-- bills table row created at reception for an insurance patient:
{
  "tenant_id":              "<hospital uuid>",
  "patient_id":             "<patient uuid>",
  "visit_id":               "<visit uuid>",
  "bill_type":              "opd",
  "payer_type":             "insurance",
  "payment_mode":           "credit",
  "status":                 "pending",
  "registration_fee":       50,
  "consultation_fee":       250,
  "total_amount":           300,
  "final_amount":           300,
  "insurance_claim_status": "pre_auth_pending",
  "patient_due":            300         -- GENERATED column: final_amount - 0 = 300
  -- All other insurance columns are NULL at this point
  -- Accountant will fill them in Insurance Cycles Edit modal
}

The patient_due column is a PostgreSQL GENERATED ALWAYS AS STORED column. It equals final_amount when no insurance has been approved yet, and automatically reduces as the accountant enters the insurance_approved_amount.

4. Pharmacy Dispensing (dispensaryPOS.html)

4.1 Normal Dispensing (Self-Pay)

When the doctor finishes a consultation and sends a prescription, it appears in the pharmacist's queue in real time. Here is the standard dispensing flow:

  1. 1
    Open dispensaryPOS.html — log in as pharmacist. The prescription queue panel on the left shows all pending prescriptions for today, updating in real time.
  2. 2
    Click the patient's name — the prescription items load on the right side. You can see medicine name, dosage, frequency, and duration.
  3. 3
    Review medicines in cart — quantities are pre-filled from the prescription. Adjust if needed (e.g., only partial stock available). You can see current stock levels and expiry dates.
  4. 4
    Select payment mode — Cash / UPI / Card. Confirm the total amount shown.
  5. 5
    Click "Dispense & Generate Bill" — stock quantities are deducted from the inventory table, a pharmacy bill is created in the bills table (bill_type = 'pharmacy'), and the prescription status changes to dispensed. A printable invoice is available.
Low stock alert If a medicine's stock_quantity falls below floor(max_stock / 2), a low stock alert badge appears next to the item. The pharmacist can still dispense but should inform the purchase department.

4.2 Insurance Patient Dispensing

When the visit's bill has payer_type ≠ 'self_pay', the pharmacy queue shows a special indicator and the payment flow changes automatically.

  1. 1
    In the prescription queue, an insurance patient's row shows a purple 🏥 Ins badge next to their name. This is populated from a map built when the queue loads (_rxPayerMap in the JS).
  2. 2
    Click the patient's name — the prescription loads as normal. Review the items and quantities.
  3. 3
    Payment mode is automatically set to Credit/Due and the selector is dimmed. The pharmacist cannot override this. An advisory notice is shown: "Insurance patient — no cash collection. Bill will be reconciled by accounts."
  4. 4
    Click "Dispense & Generate Bill" — medicines are dispensed, stock is deducted from inventory, and a pharmacy bill is created with payment_mode = 'credit' and status = 'pending'. No cash is collected at the counter.
Critical — Do NOT manually override the payment mode for insurance patients The pharmacy bill will be reconciled by the accounts department as part of the insurance claim cycle. Overriding the payment mode to Cash/UPI creates a data inconsistency — the OPD bill will show insurance payer type but the pharmacy bill will show self-pay, breaking the insurance claim register.

If the page is reloaded mid-session, the _rxPayerMap is rebuilt automatically when loadQueue() runs. If badges are missing after a reload, hard-refresh with Ctrl+Shift+R and wait for the queue to fully load.

5. IPD Discharge & Insurance Settlement (ipd.html)

5.1 Medical Discharge (Doctor / Nurse)

The medical portion of a discharge is handled by the treating doctor or a nurse. This captures clinical information and changes the bed status.

  1. 1
    Open ipd.html — the admission list shows all currently admitted patients. Find the patient being discharged.
  2. 2
    Click "Discharge" next to the patient's name — a discharge drawer slides in from the right.
  3. 3
    Select discharge type: Discharged (normal) / LAMA (Left Against Medical Advice) / Transferred (to another facility) / Deceased. Each type is recorded in the ipd_admissions.status column.
  4. 4
    Enter discharge date and time — defaults to now. Can be backdated if the documentation is being completed retrospectively.
  5. 5
    Complete the discharge summary — diagnosis at discharge, condition at discharge, follow-up instructions, and diet advice are filled in.
  6. 6
    If the patient is self-pay, proceed directly to Confirm Discharge. The bed status changes to Vacant automatically.

5.2 Insurance Settlement at Discharge

If the admitted patient has an associated bill with payer_type ≠ 'self_pay' (detected automatically by the system), the discharge drawer automatically shows an additional Insurance Settlement Panel below the discharge summary form.

How the system detects insurance patients When the discharge drawer opens, the system queries the bills table for any bill linked to this admission with payer_type != 'self_pay'. If found, the Insurance Settlement Panel appears. If not found (self-pay), the panel is hidden.

The Insurance Settlement Panel contains:

FieldWhat to EnterDatabase Column
Approved Amount (₹) The amount the TPA has verbally or in writing approved to pay. Enter 0 if not yet approved — claim status will remain submitted. insurance_approved_amount
Claim Status Defaults to "Submitted to TPA ✓". Change to pre_auth_approved if TPA has approved but not yet received documents. insurance_claim_status
Patient Due (calculated) Read-only. Updates live as you type the Approved Amount. = Total Bill − Approved Amount. patient_due (GENERATED column — read only)
Collect patient due now Tick this checkbox if the patient is paying their share before leaving. Triggers payment mode selector

The system validates that insurance_approved_amount does not exceed final_amount. If you enter more than the bill total, an inline error message blocks the save: "Approved amount cannot exceed total bill amount."

5.3 If Patient Due Is Collected at Discharge

When the "Collect patient due now" checkbox is ticked:

  1. 1
    A payment mode selector appears (Cash / UPI / Card). Select the mode used.
  2. 2
    The patient due amount (e.g., ₹1,200) is shown as the amount to collect. Confirm with the patient.
  3. 3
    Click "Confirm Discharge" — the system records the collection and updates the bill:
    • bill.status → 'partial' (patient portion paid; TPA portion pending)
    • insurance_claim_status → 'submitted' (claim filed with TPA)
    • patient_due → reflects the remaining TPA outstanding amount (the approved amount the TPA has agreed to pay but not yet sent)

If the "Collect patient due now" checkbox is left unticked, the bill status remains 'pending' and the accountant must handle collection separately through the Finance dashboard.

Bill status meanings after discharge:

Bill StatusMeaningWho Resolves
pendingNothing paid yet — patient portion not collected, TPA not settledAccountant follows up
partialPatient share collected; TPA payment awaitedAccountant tracks TPA wire
paidFully paid — either all self-pay collected, or TPA settled in full + patient share collectedNo action needed
cancelledBill voided (super_admin only)N/A

6.3 Insurance Cycles Group

This group is visible to all roles (including receptionists in Insurance Operations Mode). It contains two sub-tabs.

Claims Register sub-tab

Shows all bills where payer_type ≠ 'self_pay'. This is the accountant's primary working view for managing insurance claims.

The Edit button for each bill opens a full 16-field modal covering:

  • Provider & TPA: Insurance company name, TPA name
  • Policy details: Policy number, member ID
  • Pre-auth: Pre-auth reference number, pre-auth status (not required / pending / approved / rejected), pre-auth amount
  • Settlement: Approved amount, settlement date, settlement amount
  • Cashless: Is cashless checkbox, non-payable amount, co-payment percentage
  • PMJAY-specific: Package code, MO approved checkbox
  • Claim status: Dropdown with all valid states (see state machine in Section 8.2)

Pre-Auth Queue sub-tab

Filtered view showing only bills with insurance_claim_status IN ('pre_auth_pending', 'pre_auth_approved'). This is the follow-up queue — bills waiting for TPA authorisation before treatment proceeds.

Morning routine for accountants Open finance.html → Insurance Cycles → Pre-Auth Queue. Review all claims that are more than 3 days old and have not yet received TPA approval. Call the TPA reference number for each and update the status after the call. Aim to have no claim in pre_auth_pending for more than 5 working days.

7. Insurance Claims Register (insurance-claims.html)

This is the dedicated page for complete insurance claim lifecycle management. Access: accountant, dept_admin, super_admin. It has four tabs.

7.1 Active Claims Tab

KPI Cards

Four cards at the top provide instant visibility into the insurance portfolio:

  • Total Active Claims — count of all non-settled, non-cancelled insurance bills
  • Total Billed (₹) — sum of final_amount across all active claims
  • Total Approved (₹) — sum of insurance_approved_amount (shows how much TPA has committed)

Filter Combinations

Use the Status filter (dropdown) and Payer filter (checkboxes) together for precise views. Examples:

  • Status: pre_auth_pending — shows all claims awaiting pre-authorisation (action required today)

  • Status: rejected — candidates for write-off requests or re-submission

When a TPA wire transfer arrives for multiple patients (common with large TPAs), use this tab to identify and record the settlement.

The tab shows provider cards — one card per insurance company / TPA — automatically grouped from the actual bill data. Each card shows the provider name, number of open claims, and total outstanding.

Click a provider card to expand it and see all claims for that TPA. The expanded view shows a settlement summary row:

7.3 PMJAY Codes Tab

PMJAY billing requires assigning an AYUSH Health Benefit Package (HBP 2.0) code to each admission. This tab provides the full list of 17 AYUSH HBP codes and allows you to assign them to specific patient bills.

Category Filter Pills

Click a category pill to filter the code list: Panchakarma / Consultation / Yoga / Naturopathy / Siddha / Unani. This helps locate the correct code quickly.

"Assign to Claim" Workflow

  1. 1
    Find the package code for the treatment being claimed (e.g., AB-AYUSH-P02 for Virechana Panchakarma, ₹6,000).
  2. 2
    Click "Assign to Claim" next to the code — a search box appears.
  3. 3
    Search for the patient by name or UHID — matching bills with payer_type = 'pmjay' appear.
  4. 4
    Select the bill and click Save — the pmjay_package_code column is updated in the bills table.
  5. 5
    Get Medical Officer approval — in the Insurance Cycles Edit modal for this bill, tick the pmjay_mo_approved checkbox after the treating doctor authorises the package.
  6. 6
    Submit to NHA PMJAY portal — log into the PMJAY portal separately and submit the claim with the package code and MO approval reference.
Warning — Verify PMJAY package rates before billing The package codes and amounts shown are based on AYUSH HBP 2.0 at the time of this build. The NHA updates Health Benefit Package rates periodically. Always verify the current approved rates at pmjay.gov.in before submitting a claim to ensure the billed amount matches the current NHA-approved rate. Incorrect rates cause claim rejection.

The aging report groups all outstanding bills by age and shows two metrics per bucket:

Bad Debt Register — Write-off Workflow

When a claim is unrecoverable (TPA has rejected it, or the patient has become unreachable), the accountant can request a write-off. This is a two-step process requiring super_admin approval.

  1. 1
    Accountant clicks "Request Write-off" FAB (floating action button, bottom-right of the Aging tab). A modal opens.
  2. 2
    Search patient by name — matching bills with outstanding patient_due appear in a results list. Select the bill to be written off.
  3. 3
    Enter write-off details:
    • Write-off amount — must be ≤ patient_due. Enter the exact amount to be written off.
    • Reason — select from dropdown: TPA Dispute / Patient Deceased / Policy Expired / Claim Rejected / Bad Debt / Other
    • Notes — free text for internal reference (e.g., TPA rejection letter reference number)
  4. 4
    Submit — a row is created in the insurance_write_offs table with status = 'pending'. The request appears in the Pending Approval sub-tab.
  5. 5
    super_admin opens the Pending Approval sub-tab, reviews the write-off details, and clicks Approve or Reject.
  6. 6
    On Approve — the database trigger trg_write_off_approval fires automatically:
    • insurance_approved_amount increases by the write-off amount
    • Since patient_due = final_amount − insurance_approved_amount, patient_due reduces to 0
    • insurance_claim_status → 'partial_settled'
    • bill.status → 'paid'
    • insurance_write_offs.approved_at gets a timestamp

The Approved Write-offs sub-tab maintains a permanent audit trail: write-off amount, reason, requested by, approved by, and timestamps. This serves as the Bad Debt Register for audits.

Write-off approval — super_admin and finance_manager only Write-off approval is restricted to super_admin and finance_manager roles. Accountants and dept_admins can view pending write-offs and submit requests but cannot approve them. This is a financial control measure — the finance_manager role exists specifically so a CFO/Finance Head can approve write-offs without having super_admin access to HR and system settings.

8. Technical Reference

8.1 Database Schema — bills Table (Insurance Columns)

8.2 insurance_claim_status State Machine

The insurance_claim_status column follows a strict progression. Jumping states is allowed but discouraged — always follow the natural sequence below for clean audit trails.


 [Reception creates insurance bill]
           │
           ▼
   pre_auth_pending          ← Initial state for all non-self_pay bills
           │
           │  TPA issues pre-auth approval letter
           ▼
   pre_auth_approved         ← Pre-auth ref number entered by accountant
           │
           │  Accountant sends claim documents to TPA
           ▼
      submitted              ← After discharge, documents filed with TPA
           │
           │  TPA pays partial amount
           ├──────────────────────────────────────────────────────┐
           ▼                                                      │
   partial_settled           ← Also set automatically when       │
           │                    a write-off is approved           │
           │  TPA pays full balance                               │
           ▼                                                      │
        settled   ◄──────────────────────────────────────────────┘
                                (full payment received)

   At any point → rejected   (TPA refuses the claim)
   On rejection  → Request Write-off → partial_settled
                                      (bad debt recorded)
not_applicable state The value not_applicable is reserved exclusively for self-pay bills (payer_type = 'self_pay'). The constraint chk_insurance_workflow_sync enforces this pairing — see Section 8.7. Never set insurance_claim_status to not_applicable for any non-self_pay bill.

8.3 patient_due Generated Column

The patient_due column is a PostgreSQL GENERATED ALWAYS AS STORED column. It is computed automatically by the database engine and cannot be written to directly. Any attempt to include it in an INSERT or UPDATE will result in an error.

-- Column definition (for reference — already applied in migration):
patient_due NUMERIC GENERATED ALWAYS AS (
    COALESCE(final_amount, 0) - COALESCE(insurance_approved_amount, 0)
) STORED

-- Reading patient_due (normal SELECT — works fine):
SELECT id, patient_due FROM bills WHERE id = '<bill_id>';

-- WRONG — this will throw: "column patient_due is a generated column"
UPDATE bills SET patient_due = 0 WHERE id = '<bill_id>';

-- CORRECT — to reduce patient_due to zero (write-off scenario):
-- Increase insurance_approved_amount to equal final_amount
UPDATE bills
SET insurance_approved_amount = COALESCE(insurance_approved_amount, 0) + <write_off_amount>
WHERE id = '<bill_id>';
-- patient_due will automatically recalculate to 0 (or lower if write_off > remaining due)

-- CORRECT — when TPA pays full approved amount:
UPDATE bills
SET
    insurance_settled_amount   = <wire_amount>,
    insurance_settlement_date  = CURRENT_DATE,
    insurance_claim_status     = 'settled',
    status                     = 'paid'
WHERE id = '<bill_id>';

8.4 insurance_write_offs Table

This table must be created via a SQL migration before the write-off features in insurance-claims.html become functional. If this table is missing, the Aging tab shows a migration banner prompting the super_admin to run the migration.

CREATE TABLE insurance_write_offs (
    id             uuid        DEFAULT gen_random_uuid() PRIMARY KEY,
    tenant_id      uuid        NOT NULL REFERENCES tenants(id),
    bill_id        uuid        NOT NULL REFERENCES bills(id),
    amount         numeric(12,2) NOT NULL CHECK (amount > 0),
    reason         text        NOT NULL,
    notes          text,
    status         text        DEFAULT 'pending'
                               CHECK (status IN ('pending','approved','rejected')),
    requested_by   uuid        REFERENCES profiles(id),
    approved_by    uuid        REFERENCES profiles(id),
    requested_at   timestamptz DEFAULT now(),
    approved_at    timestamptz
);

ALTER TABLE insurance_write_offs ENABLE ROW LEVEL SECURITY;

CREATE POLICY "rls" ON insurance_write_offs
    FOR ALL TO authenticated
    USING (tenant_id = (SELECT tenant_id FROM profiles WHERE id = auth.uid()))
    WITH CHECK (tenant_id = (SELECT tenant_id FROM profiles WHERE id = auth.uid()));

GRANT ALL ON insurance_write_offs TO authenticated;

CREATE INDEX idx_write_offs_bill
    ON insurance_write_offs (bill_id);

CREATE INDEX idx_write_offs_status
    ON insurance_write_offs (tenant_id, status)
    WHERE status = 'pending';

8.5 Write-off Approval Trigger

When a super_admin approves a write-off (sets insurance_write_offs.status to 'approved'), this trigger fires automatically to update the linked bill.

CREATE OR REPLACE FUNCTION process_approved_write_off()
RETURNS TRIGGER AS $$
BEGIN
    -- Only fires when status transitions from 'pending' to 'approved'
    IF NEW.status = 'approved' AND OLD.status = 'pending' THEN
        UPDATE bills
        SET
            -- Increase insurance_approved_amount → patient_due (GENERATED) reduces
            insurance_approved_amount = COALESCE(insurance_approved_amount, 0) + NEW.amount,
            -- Mark claim as partially settled (not full settlement — TPA rejected part)
            insurance_claim_status    = 'partial_settled',
            -- Close the bill — patient_due is now 0
            status                    = 'paid'
        WHERE id = NEW.bill_id;

        -- Stamp the approval timestamp on the write-off record
        NEW.approved_at := now();
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER trg_write_off_approval
    BEFORE UPDATE ON insurance_write_offs
    FOR EACH ROW
    EXECUTE FUNCTION process_approved_write_off();
Critical — SECURITY DEFINER is required The trigger function uses SECURITY DEFINER to run as the table owner, bypassing RLS. This is required because the trigger must UPDATE the bills table — which belongs to the authenticated user's tenant context — from within the trigger's execution context. Without SECURITY DEFINER, the RLS policy on bills blocks the UPDATE (the trigger runs in a different user context). Do not remove SECURITY DEFINER from this function.

8.6 RLS Policy Pattern

All tables in AyurXpert follow the same RLS pattern. The tenant isolation is always resolved through the profiles table, never from JWT claims.

-- Standard RLS pattern used across ALL tables in this project:
CREATE POLICY "rls" ON <table_name>
    FOR ALL TO authenticated
    USING (
        tenant_id = (SELECT tenant_id FROM profiles WHERE id = auth.uid())
    )
    WITH CHECK (
        tenant_id = (SELECT tenant_id FROM profiles WHERE id = auth.uid())
    );
Critical — Never use JWT app_metadata for tenant_id Do NOT use auth.jwt() -> 'app_metadata' ->> 'tenant_id' in any RLS policy. AyurXpert stores tenant_id in the profiles table, NOT in JWT app_metadata. Using the JWT expression returns NULL for all users, effectively making the table visible to no one (or to everyone, depending on how the policy is written). The subquery (SELECT tenant_id FROM profiles WHERE id = auth.uid()) is the only correct pattern for this codebase.
anon role access — special case If a table needs to be readable by unauthenticated users (e.g., the tenants table for the registration page), two things are required together: (1) GRANT SELECT ON <table> TO anon AND (2) a separate RLS policy allowing anon access. Either one alone is insufficient. A missing GRANT returns HTTP 400 from PostgREST (not 403).

8.7 Constraint: chk_insurance_workflow_sync

This constraint enforces consistency between payer_type and insurance_claim_status. It prevents the common error of creating a bill with payer_type = 'insurance' but accidentally leaving claim_status as 'not_applicable'.

-- Applied to the bills table during Session 65 migration:
CONSTRAINT chk_insurance_workflow_sync CHECK (
    -- Self-pay bills MUST have 'not_applicable' claim status
    (payer_type = 'self_pay' AND insurance_claim_status = 'not_applicable')
    OR
    -- All non-self-pay bills MUST NOT have 'not_applicable' claim status
    (payer_type <> 'self_pay' AND insurance_claim_status <> 'not_applicable')
)

-- This constraint will REJECT any of these combinations:
-- ✗ payer_type = 'insurance', insurance_claim_status = 'not_applicable'
-- ✗ payer_type = 'self_pay', insurance_claim_status = 'pre_auth_pending'
-- ✗ payer_type = 'pmjay', insurance_claim_status = 'not_applicable'

-- These combinations are ALLOWED:
-- ✓ payer_type = 'self_pay', insurance_claim_status = 'not_applicable'
-- ✓ payer_type = 'insurance', insurance_claim_status = 'pre_auth_pending'
-- ✓ payer_type = 'pmjay', insurance_claim_status = 'submitted'
Constraint error at bill save? If the application throws a constraint violation when saving a bill, check two things: (1) if payer_type is being set to a non-self_pay value without setting insurance_claim_status to something other than 'not_applicable'; (2) if an old bill's payer_type was changed from insurance to self_pay without also resetting insurance_claim_status to 'not_applicable'.

9. Common Scenarios (End-to-End)

Scenario A — Regular Self-Pay OPD Patient

The most common scenario. Patient walks in for an OPD consultation and pays cash.

  1. Receptionist searches patient by phone number in reception.html.
  2. Patient found (or new patient form filled). OPD selected: General OPD. Doctor selected: Dr. Sharma.
  3. Fees auto-fill: Registration ₹50 + Consultation ₹200 = Total ₹250.
  4. Payer Type: Self-Pay (default). Payment Mode: Cash. Status: Paid.
  5. Click "Register & Create Visit" → Token #8 issued. Doctor's queue updated.
  6. Pharmacist receives prescription from doctor. Dispenses medicines. Collects ₹180 cash for medicines.
  7. Finance dashboard shows: Revenue Today +₹430. No entries in Insurance Cycles.
  8. End of day: both bills (OPD + pharmacy) are status = 'paid'. Zero outstanding.

Scenario B — Insurance TPA Patient (OPD)

Patient has corporate insurance through their employer. TPA: Medi Assist India.

  1. [Reception] Receptionist selects 🏥 Insurance/TPA. No cash collected. Token issued. Bill created: payer_type = 'insurance', status = 'pending', insurance_claim_status = 'pre_auth_pending'.
  2. [Pharmacy] Purple 🏥 Ins badge shown. Pharmacist dispenses without cash collection. Pharmacy bill: payment_mode = 'credit'.
  3. [Accountant — finance.html Insurance Cycles] Opens the bill in Edit modal. Fills: insurance_provider = "Star Health", tpa_name = "Medi Assist India", policy_number = "MAI-00234567". Saves. insurance_claim_status still 'pre_auth_pending' for now (no pre-auth required for OPD at this hospital).
  4. Accountant changes insurance_claim_status → 'submitted' (documents submitted to TPA). Bill status remains 'pending'.
  5. [TPA pays — 15 days later] TPA wire transfer of ₹320 received (covers consultation + partial medicines). Accountant opens finance.html → Insurance Cycles → Edit modal. Fills: insurance_approved_amount = ₹320, insurance_settled_amount = ₹320, insurance_settlement_date = today. insurance_claim_status → 'settled'. Bill status → 'paid'.
  6. patient_due recalculates: ₹430 − ₹320 = ₹110. Accountant contacts patient to collect the ₹110 co-payment (if applicable per policy).
  7. Finance dashboard: Insurance Claims count decreases by 1. Revenue This Month increases.

Scenario C — PMJAY Panchakarma IPD Patient (Most Complex)

Patient is a PMJAY beneficiary admitted for Virechana Panchakarma therapy for 7 days. This is the most complex financial lifecycle in the system.

  1. [Reception — reception.html] Verify patient's PMJAY e-card physically. Select 🟢 PMJAY payer type. Token issued. Bill: payer_type = 'pmjay', insurance_claim_status = 'pre_auth_pending'.
  2. [IPD admission — ipd.html] Doctor creates IPD admission. Selects ward and bed (2-step bed picker). Diagnosis entered. Panchakarma therapy sessions scheduled in therapist.html.
  3. [PMJAY package assignment — insurance-claims.html PMJAY Codes tab] Accountant finds package code: AB-AYUSH-P02 (Virechana Panchakarma, ₹6,000). Clicks "Assign to Claim" → searches patient → selects PMJAY bill → saves. pmjay_package_code updated in bills table.
  4. [MO approval] Treating doctor (Medical Officer) reviews the package assignment. Accountant ticks pmjay_mo_approved checkbox in Insurance Cycles Edit modal. insurance_claim_status → 'pre_auth_approved'.
  5. [Submit to NHA PMJAY portal] Accountant logs into the NHA PMJAY portal externally, submits claim with package code AB-AYUSH-P02. Claim reference obtained from NHA portal.
  6. [IPD discharge — ipd.html] Doctor fills discharge summary. Insurance Settlement Panel shows automatically (payer_type = 'pmjay'). Approved Amount: ₹6,000. Collect patient due now: unchecked (cashless). insurance_claim_status → 'submitted'. Bill status → 'partial'.
  7. [TPA settlement] NHA / State PMJAY agency processes payment. Accountant receives wire of ₹6,000. Finance → Insurance Cycles → Edit: insurance_settled_amount = ₹6,000, settlement_date = today. insurance_claim_status → 'settled'. Bill status → 'paid'.
  8. patient_due = ₹6,000 − ₹6,000 = ₹0. Fully settled. No patient collection needed.
PMJAY cashless important note When the is_cashless checkbox is ticked for a PMJAY bill, the system confirms that no patient payment will be collected at any stage. The total bill amount must match the approved package code amount. Any excess billing is prohibited under PMJAY scheme rules.

Scenario D — Write-off / Bad Debt

Insurance claim submitted 100 days ago. TPA has rejected the claim. Patient is now unreachable. Accountant needs to write off the outstanding amount.

  1. [Identify the claim] Accountant opens insurance-claims.html → Aging & Write-offs tab. The bill appears in the red bucket (>90 days). insurance_claim_status = 'rejected'. patient_due = ₹4,500.
  2. [Request write-off] Accountant clicks the "Request Write-off" FAB. Searches patient name "Raman Pillai". Selects the rejected bill. Enters: amount = ₹4,500, reason = "TPA Dispute / Rejected", notes = "Rejection letter TPA-REF-2026-0341 received 10 June 2026. Patient unreachable after 3 attempts." Submits.
  3. Write-off record created: status = 'pending'. Appears in Pending Approval sub-tab.
  4. [Super Admin reviews] Super Admin opens insurance-claims.html → Aging tab → Pending Approval sub-tab. Reviews the write-off details and rejection letter reference. Decides it is valid. Clicks "Approve".
  5. [Database trigger fires] trg_write_off_approval runs:
    • bills.insurance_approved_amount += ₹4,500 → now = ₹4,500
    • patient_due (GENERATED) = ₹4,500 − ₹4,500 = ₹0
    • bills.insurance_claim_status = 'partial_settled'
    • bills.status = 'paid'
    • insurance_write_offs.approved_at = now()
  6. The Approved Register shows this write-off permanently with the ₹4,500 amount, reason, and both accountant + super_admin names with timestamps. Used for income tax write-off documentation in annual accounts.
  7. Finance dashboard: Outstanding >30d KPI card count decreases. Revenue dashboard reflects the bad debt adjustment.

Check fee-admin.html — find the fee, verify its status and the OPD it is linked to. Ask super_admin to activate if needed. Confirm the OPD selected at reception matches the OPD in the fee record.
Insurance Settlement Panel not showing in IPD discharge drawer The system could not find a bill linked to this admission with payer_type ≠ 'self_pay'. Either the bill was not linked to the visit, or payer_type was incorrectly saved as 'self_pay'. Check the bills table for the patient: confirm visit_id or patient_id links to this admission, and confirm payer_type is set correctly. Update via Supabase Dashboard if needed.
Write-off approved but patient_due did not change to 0 The trigger trg_write_off_approval may not have been created, or the write-off amount entered was less than the actual patient_due. Check if the trigger exists: run SELECT trigger_name FROM information_schema.triggers WHERE trigger_name = 'trg_write_off_approval'; in SQL Editor. If missing, re-run the Phase 3 migration. If amount was less than patient_due, a second write-off request may be needed for the remaining balance.

Run: GRANT ALL ON bills TO authenticated; in SQL Editor. PostgREST returns HTTP 400 (not 403) when the table-level GRANT is missing — this is a known Supabase/PostgREST behaviour. 403 is returned only when the GRANT exists but RLS blocks the row.
